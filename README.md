## Опис застосунку

Проєкт є інформаційною системою продажу залізничних квитків, яка дозволяє працювати з даними про потяги, станції, маршрути та квитки в єдиній базі даних. Система забезпечує перегляд списку потягів, перегляд станцій, через які проходить конкретний потяг, а також пошук маршрутів між двома станціями без пересадок. Окрім цього реалізовано облік рейсів і проданих квитків, включаючи підтримку тимчасових рейсів, які можуть виконуватись сезонно або у святкові періоди.

### Типові запити системи:
- список потягів, що проходять через певну станцію;
- список станцій конкретного потяга;
- пошук маршрутів між двома станціями без пересадок.

### Розширений функціонал:
- визначення топ-5 станцій з найбільшим трафіком потягів;
- перегляд тимчасових рейсів за поточний тиждень;
- пошук маршрутів з обмеженою кількістю пересадок.

```text

    TRAIN_RUNS {

    }
```

## Правила комітів

У проєкті використовується узгоджена система комітів. Основні типи комітів:

- **feat** — новий функціонал  
- **fix** — виправлення помилок  
- **ui** — зміни інтерфейсу  
- **db** — зміни бази даних або SQL  
- **refactor** — покращення структури коду без зміни логіки  
- **docs** — документація  
- **test** — тести  

Коміти повинні бути атомарними та зрозумілими. Бажано використовувати англійську мову та писати короткі, але змістовні повідомлення.



## Правила роботи з гілками та злиття (Git Flow)

У проєкті використовується класична модель Git Flow. Основні гілки: **main** (стабільна версія) та **develop** (основна розробка).

### Додаткові гілки:
- feature/* — новий функціонал  
- fix/* — виправлення помилок  
- ui/* — зміни інтерфейсу  

### Процес роботи:
створюється окрема гілка від develop, у ній виконується розробка, після чого створюється Pull Request у develop. Кожен Pull Request повинен проходити рев’ю хоча б одного учасника команди. Злиття виконує Team Lead. Прямі зміни в main заборонені.
